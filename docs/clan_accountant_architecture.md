## Назначение и управление

Проект представляет собой многоцелевую аналитическую систему для игры Mech Heroes, решающую три задачи:
1.  **Мониторинг клана («Казначей клана»):** Отслеживание прогресса и активности членов клана ОРДА.
2.  **Глобальная аналитика Арены:** Слежение за Топ-50 игроками мира, эволюцией их отрядов и рейтингов.
3.  **Аналитика боёв и личный прогресс:** Детальный разбор сражений и ведение досье на оппонентов для игрока `ksotar`.

### 🚨 Безопасность сессии и флаг `--force`
**ВНИМАНИЕ:** Флаг `--force` отключает механизмы защиты сессии, что гарантированно приводит к потере соединения с игрой. Использование этого флага в любых скриптах или автоматизациях **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО**. Применение возможно только в исключительных ситуациях после получения прямого, явного разрешения от владельца системы.

Для автономной работы системы по расписанию используется файл **`run_accountant.bat`** в корне проекта. Он выполняет роль «дирижёра» всей системы:

3.  **Полный цикл:**
    *   **Клановый этап:** `clan_monitor/clan_accountant.py` (сбор данных клана).

## Архитектура: поток данных

```
[Запуск скрипта]
       │
       ▼
  fetch_data()
  ├─ POST /init (1)      → получение sessionID, LastCommandId
  ├─ POST /commands (3)  → UseServiceCommand: RefreshArenaLeaderboards
  ├─ [Пауза 5 секунд]    → ожидание сброса серверного кэша
  ├─ POST /init (2)      → получение АКТУАЛЬНЫХ данных клана
  └─ POST /directcommand → GetUsersRawInfos → данные профилей всех членов клана
       │
       ▼
  generate_web_report()
  ├─ Обновляет members_name_db.json (ники, роли, аватары)
  ├─ Сохраняет profile_history.json → актуальный рейтинг, клан и ник игрока (независимо от отряда)
  ├─ Сохраняет снэпшот очков → snapshots/points_utc_YYYY-MM-DD_HH-MM.json
  ├─ Читает ВСЕ снэпшоты → строит понедельные периоды
  ├─ Считает прирост по «Алгоритму Истины»
  ├─ Генерирует HTML-файлы отчётов → clan/ORDA/reports/report_YYYY_WNN.html
  └─ Генерирует clan/ORDA/index.html (редирект на последний отчёт)
       │
       ▼
  run_git_push()  (если AUTO_PUSH = True)
  └─ git add -A → git commit → git push
```

## Файлы и директории

### Входные / конфигурационные

| Файл | Описание |
|---|---|
| `config.json` | `USER_ID`, `AUTH_KEY`, `VERSION` игры. Обновляется ботом автоматически при смене версии. |
| `manual_adjustments.json` | Ручные корректировки очков. Структура: `{"YYYY-MM-DD": {"userId": [очки_при_выходе]}}`. Нужны для игроков, покинувших клан в течение дня — сервер при следующем запросе уже не вернёт их очки. |
| `members_name_db.json` | Накопительная база ников, ролей и аватаров всех участников. Ключ — `userId` (строка). Пишется при каждом запуске. |
| `profile_history.json` | (В папке игрока) История изменений `arenaRating`, `clanProfile` и `nickname`. Обновляется только при реальных изменениях. |
| `translations_cache.json` | Кэш переводов черт аватара (борода, очки и т.д.) — чтобы не переводить повторно. |

### Снэпшоты

**Директория:** `snapshots/`

Каждый запуск создаёт файл `points_utc_YYYY-MM-DD_HH-MM.json`:
```json
{
  "pts": {"227408": 10206, "123456": 8500, ...},
  "clanRating": 11199931
}
```
Снэпшоты **никогда не удаляются** — это сырая история. По ним строятся все отчёты.

### Выходные HTML

**Директория:** `clan/ORDA/`

| Файл | Описание |
|---|---|
| `index.html` | Редирект на последний недельный отчёт |
| `reports/report_YYYY_WNN.html` | Отчёт за конкретную ISO-неделю |

## Ключевые модули и функции

### `fetch_data() → (hier, users, rating)`

Синхронизированный цикл запросов к API:

1. **`POST /init` (RequestId: 1)** — первичная инициализация сессии, получение `LastCommandId`.
2. **`POST /commands` (RequestId: 3)** — принудительная команда `UseServiceCommand` с типом `RefreshArenaLeaderboards`. Позволяет обойти серверное кэширование и получить свежие данные по боям и очкам до планового обновления сервера.
3. **`time.sleep(5)`** — критическая задержка (5 секунд), необходимая для того, чтобы сервер успел обновить состояние `/init` после команды Refresh.
4. **`POST /init` (повторно)** — забор обновленной иерархии клана (`hier`) и актуального рейтинга.
5. **`POST /directcommand` (RequestId: 2)** — получение профилей участников (ники, аватары) через тип `GetUsersRawInfos`.

**Auto-healing версии:** если `/init` вернул error code `61058` («outdated version»),
бот автоматически парсит новую версию из текста ошибки, обновляет `config.json` и повторяет запрос.

### `generate_web_report(hier, users, current_rating)`

Основная логика отчёта. Алгоритм по шагам:

1. **Обновление базы имён** — для каждого `user` из ответа API сохраняет `nick`, `role`, переведённые черты аватара.
2. **Сохранение снэпшота** — `pts` (userId → points) + `clanRating`.
3. **Загрузка всех снэпшотов** — сортирует по времени, группирует по неделям (ключ `YYYY_WNN`)

### 4.4. Защита активной сессии (Session Safety)
Для предотвращения разрыва текущей игровой сессии пользователя (ошибка `61081: Can't find context`) бот выполняет двухэтапную проверку перед любым обращением к API:
1.  **Проверка заголовка окна:** Поиск процесса с заголовком, содержащим "Боевые роботы".
2.  **Проверка сетевых соединений:** Если окно найдено, выполняется анализ TCP-соединений (`netstat`). Если обнаружено соединение в статусе `ESTABLISHED` с IP сервера (`tanks.ya.patternmasters.ru`), бот прекращает работу.

**Переопределение:** Использование флага `--force` при запуске (`python clan_accountant.py --force`) полностью отключает эту проверку, позволяя обновить отчёт даже при запущенной игре.

4. **Расчёт прироста («Алгоритм Истины»):**
   - **Monday Base:** Для каждого понедельника (начало новой недели) база `prev_day_end` принудительно устанавливается в **0**. Это гарантирует, что весь прогресс первого дня (после еженедельного сброса в игре) идет в плюс.
   - **Reset Detection (Триггер сброса):** Для игроков, перечисленных в `manual_adjustments.json`, бот анализирует поток снапшотов за день. Сброс (обнуление очков) фиксируется, если любое последующее значение меньше предыдущего (`vals[j+1] < vals[j]`).
   - **Формула при сбросе:** `Growth = (Значение_перед_сбросом - База_дня) + Значение_на_конец_дня`. Это позволяет учесть как набитое до обнуления, так и прогресс после него.
   - **Стандартный расчет:** Если сброса не было или игрок не входит в список для специальной обработки, используется простая дельта: `Growth = Значение_на_конец_дня - База_дня`.
   - **Преемственность:** В конце каждых суток `prev_day_end` обновляется до финального значения дня для использования в качестве базы на следующие сутки.

5. **Определение присутствия:**
   - Сегодня: игрок присутствует, если он в текущей иерархии ИЛИ есть запись в `manual_adjustments`.
   - Прошлые дни: бот проверяет наличие записей в снапшотах или ручных корректировок. Уход из клана помечается в таблице символом `X`.

6. **Клановая статистика:**
   - **Исторический рейтинг:** расчет прогресса клана относительно фиксированной базы `11199931` (старт проекта на W19).
   - **Сгоревшие суммы:** Отображение разницы между личным заработком членов и фактическим ростом рейтинга клана (эмодзи 🔥). Считается как `Суммарный_прирост - Изменение_рейтинга`.

7. **Генерация HTML:** Полное восстановление оригинального дизайна (Master UI) с использованием шрифтов Orbitron и Roboto Mono, навигацией через `<nav>` и цветовым кодированием ролей.

## Стандарты точности данных (Data Parity)

Для этого проекта установлен стандарт **Absolute Truth Parity**:

1. **Единый источник истины (SSOT):** Все отчеты строятся исключительно на основе сырых JSON-снапшотов в директории `snapshots/`.
2. **Временная зона:** Все расчеты ведутся строго в **UTC**. Конвертация в MSK происходит только на этапе отображения времени последнего обновления в шапке отчета.
3. **Границы дней:** Строго 00:00:00 UTC. Никаких смещений на "последний доступный снэпшот".
4. **Математика сбросов:** Принудительный учет обнулений очков внутри суток для ключевых игроков (те, кто в `manual_adjustments.json`), что позволяет видеть реальный набитый прогресс, а не только разницу между концом и началом дня.
5. **Целостность данных:** Понедельник всегда стартует с 0, обеспечивая чистоту еженедельной статистики.
### `translate_traits_batch(traits_list)`

Переводит черты аватара (прическа, борода, очки и т.д.) с английского на русский.
Использует жёстко зашитый словарь (без обращения к внешним API). Результат кэшируется в `translations_cache.json`.
Ранее использовался `GoogleTranslator` (библиотека `deep_translator`) — сейчас не нужен, но `import` остался.

### `run_git_push()`

Простой `subprocess`-вызов `git add -A && git commit && git push`.
Управляется флагом `AUTO_PUSH = True` в начале файла.

**Заголовки (все запросы):**

## Известные ограничения и технический долг

- `import deep_translator` — импорт не нужен, перевод сейчас полностью на словаре. Можно убрать.
- Снэпшоты не очищаются автоматически — директория будет расти бесконечно.
- Нет обработки ситуации, когда `members_name_db.json` содержит userId игрока, которого нет в текущем ответе `GetUsersRawInfos` (был в клане давно).
- `manual_adjustments.json` нужно заполнять вручную перед запуском, если кто-то ушёл из клана.
