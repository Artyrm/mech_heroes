# Подсистема battle_analytics (v0.2.2)

Данная директория содержит ядро аналитики боев Мех Героев. 

## Архитектура и поток данных (Data Flow)

1. **Сбор (Fetch):** Скрипт `fetch_and_store_battles.py` обращается к API игры (`/init`), получает глобальное состояние (JSON) и сохраняет его целиком в `../init_dumps/init_YYYY-MM-DD_HH-MM-SS.json`. Из него извлекается массив `battlesHistory`. Для обеспечения гарантированной доставки свежих боев используется цикл принудительного обновления Арены (RefreshArenaLeaderboards).
2. **Экстракция и маршрутизация:** Для заданных никнеймов (target_nicks) скрипт ищет совпадения в `battlesHistory`. Найденные бои извлекаются в виде индивидуальных JSON файлов и сохраняются в `battle_analytics/{Nickname}/`. Формат имени: `battle_YYYY-MM-DD_HH-MM-SS_{battle_id}.json`.
3. **Генерация отчетов:** При сохранении нового JSON-боя автоматически вызывается `generate_html_report.py`, который парсит JSON и генерирует визуальный HTML-отчет в той же папке.
4. **Сравнительный анализ:** Утилита `generate_comparison_report.py` позволяет создать детальное сравнение двух боев (например, последней победы и последнего поражения). Она проводит параллельный аудит изменений в отрядах игрока и оппонента, включая генералов, с текстовой сводкой всех изменений.

## Использование инструментов

### Генерация обычного отчета
```bash
python generate_html_report.py <путь_к_json>
```

### Генерация сравнительного отчета (разные игроки)
```bash
python generate_custom_comparison.py <json_1> <json_2> <выходной_html> <имя_1> <имя_2>
```
Этот скрипт сравнивает отряды **врагов** из двух разных боев, что полезно для сопоставления билдов двух разных топовых игроков.

Утилита `extract_battles.py` позволяет проделать шаги 2-3 ретроспективно, просканировав все исторические дампы в папке `../init_dumps/`.

## Структура данных боя (JSON)

Каждый `battle_*.json` содержит структуру из `battlesHistory`. Ключевые узлы:
- `fightTime`: Unix timestamp боя.
- `nick`: Никнейм оппонента.
- `statistics.player.units` и `statistics.enemy.units`: Словари юнитов, где ключ — номер слота.
- Внутри каждого юнита:
  - `state`: Идентификаторы, звезды, сырые уровни (`level`), экипировка (`equipables`).
  - `statistics`: Боевые показатели (`damageDone`, `healthLost`, `killsCount`, `healthHealed`).

## Специфические алгоритмы парсинга

При написании новых отчетов или скриптов учитывайте следующие реализованные механики, необходимые для расшифровки сырых данных игры:

### 1. Расчет уровней (Level.Step)
Сырой уровень (**level**), приходящий с сервера для **Юнитов и Генералов** (например, 476), не является фактическим игровым уровнем. Каждый игровой уровень разбит на 4 шага (палочки прокачки), при этом первый уровень начинает заполняться с нулевого индекса.
- Формула: `Уровень = (raw // 4) + 1`, `Шаг = (raw % 4) + 1`.
- Ограничение: Максимальный уровень в игре — 120. Если расчетный уровень >= 120, выводится просто "120". Если `Шаг == 1`, выводится только целое значение уровня (например, "1", а не "1.1"). 
- Эта логика инкапсулирована в `generate_html_report.py` -> `format_level()`.

**Важно:** Уровни **Экипировки (Gear/Equipment)** приходят в JSON уже в готовом виде (1-120) и отображаются без пересчета и без шагов прокачки.

### 2. Определение Атаки/Защиты
Позиция игрока (нападение или защита) не передается явным флагом. Она вычисляется на основе занятых слотов:
- Атакующий всегда занимает слоты с меньшими ID (на игровой доске).
- Защитник занимает слоты с бОльшими ID.
- Алгоритм: Сравнивается `min(player_slots)` и `min(enemy_slots)`. У кого минимальное значение меньше, тот и является нападающим.

## Time Zone Policy

Для обеспечения консистентности данных во всей аналитической системе приняты следующие стандарты работы с временем:

1. **Storage & Backend (UTC):** Имена всех дампов состояний и файлов боев (`battle_*.json`) содержат дату и время события в формате **UTC** (стандарт игрового сервера). Это **единственный и исключительный** источник информации о времени события; системные атрибуты файлов (дата создания/изменения файловой системы) в аналитике не используются.
2. **Analysis (UTC):** Все скрипты обработки и вычисления временных рядов (интервалы, периоды активности, выборки за день) работают строго по временным меткам, извлеченным из названий файлов (**UTC**).
3. **Frontend & Visualization (MSK):** При выводе информации пользователю (в HTML-отчетах, дашбордах и личном досье) время преобразуется в **MSK** (UTC+3). 
   - Все временные метки в интерфейсе отображаются в формате `DD.MM HH:MM`.
   - Пороговые значения "начало дня" (для выборки периода) вычисляются на основе смещения МСК относительно временных меток из названий файлов.

При написании новых аналитических скриптов обязательно учитывайте, что исходное время берется строго из названий файлов, и не применяйте смещение времени до этапа финальной визуализации.
