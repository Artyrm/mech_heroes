# Расследование: Причины задержки обновления истории боёв (May 21-22, 2026)

Данный документ фиксирует установленные факты из кода и наблюдаемое поведение системы в период сбоя обновления аналитики боёв.

---

## 1. Установленные факты из кода

### Факт А: Иерархия выполнения (Оркестрация)
Согласно `run_accountant.bat`, выполнение всегда идет в строгом порядке:
1. `clan_monitor\clan_accountant.py` (всегда делает запрос `/init`, создаёт `init_dumps/init_{timestamp}.json`).
2. `arena_update.py` (вызывает `fetch_arena.py`, затем `fetch_and_store_battles.py`).

### Факт Б: Механизм "Оптимизации" (Доверие дампу)
В скриптах `fetch_arena.py` и `fetch_and_store_battles.py` реализована логика:
*   Если в папке `init_dumps/` найден файл, созданный менее 15 минут назад, скрипт **не делает запрос к API**.
*   Вместо этого он считывает данные из этого файла.
*   *Следствие:* Эти скрипты в автоматическом режиме всегда работают с данными, которые получил `clan_accountant.py` несколькими секундами ранее.

### Факт В: "Волшебная команда" Refresh
В скрипте `arena/fetch_arena.py` (строка 192) выполняется `directcommand` с типом **`RefreshArenaLeaderboards`**.
*   Эта команда отправляется **после** того, как скрипт обработал данные из `/init` (или из дампа).
*   В `fetch_and_store_battles.py` аналогичных команд рефреша нет — он только читает `battlesHistory`.

### Факт Г: Логика "Новизны" в боях
В `fetch_and_store_battles.py` (строка 144) проверка на новый бой осуществляется исключительно по наличию файла. Если файл `battle_{fightTime}.json` уже существует, скрипт игнорирует запись.

### Факт Д: Условный второй запрос /init
В коде `fetch_and_store_battles.py` (строка 95) присутствует полноценный вызов API `/init`.
*   **В штатном режиме:** Этот вызов никогда не происходит, так как скрипт прерывается на строке 66 из-за "свежего" дампа Казначея.
*   **В режиме --force:** Блок переиспользования дампа игнорируется (`if not force_run`), и скрипт выполняет **второй за цикл** запрос к API.
*   **Следствие:** Именно этот второй запрос, идущий *после* команды Refresh от Арены, является единственным источником свежих данных в текущей архитектуре.

## 2. Гипотезы и интерпретации (Требуют проверки)

### Гипотеза №1: Задержка серверного кэша (Snapshot Lag)
**Суть:** Запрос `/init`, который делает Казначей, возвращает "холодный" слепок состояния игрока. В этом слепке очки клана уже обновлены, но `battlesHistory` содержит данные на момент последнего выхода игрока из сети или последнего серверного снапшота.
**Почему это важно:** Если это так, то Казначей всегда будет получать "тухлую" историю боев, а скрипт боев, доверяя его дампу, никогда не увидит свежих матчей.

### Гипотеза №2: Refresh работает только для текущей сессии
**Суть:** Команда `RefreshArenaLeaderboards` обновляет историю боев на сервере только для того `sessionID`, внутри которого она была послана. 
*   Поскольку автоматика каждый час создает **новые сессии** через `/init`, команда Refresh из "Сессии А" не влияет на то, что увидит Казначей в "Сессии Б" через час.
**Почему это важно:** Это объясняет, почему данные не обновлялись "к следующему автозапуску".

### Гипотеза №3: Эффективность ручного запуска --force
**Суть:** Флаг `--force` в `fetch_and_store_battles.py` заставляет его игнорировать дамп Казначея и идти в API самостоятельно.
*   Это происходит **после** того, как скрипт Арены уже послал команду `Refresh`.
*   Второй запрос `/init` (сделанный вручную) получает данные, которые сервер успел подготовить после "пинка" от Арены.

## 3. План верификации

1.  **Тест "Двойной /init":** Модифицировать скрипт боев, чтобы он делал `/init`, затем паузу 5 секунд, затем второй `/init`. Сравнить поле `battlesHistory` в первом и втором ответе.
2.  **Аудит сессий:** Проверить, сохраняется ли `sessionID` между вызовами Арены и Боев. (Текущий факт: Арена пишет его в `arena/session.json`, но Бои его не читают).
3.  **Логирование Refresh:** Добавить в лог подтверждение от сервера на команду `RefreshArenaLeaderboards` и проверить, меняется ли после этого `lastUpdateTime` в Арене.
